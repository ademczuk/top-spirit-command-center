Wave 1 — Power BI Desktop pilot: build pack & BICC handover runbook

The BICC's first self-service report, built from file exports only — no live connectors and no new licence spend. AI/automation may draft measures and lineage notes; a named human (Finance SME and BICC) approves every definition and reconciles every figure before the report is trusted. Stays inside Microsoft 365 E3.

Project-team working preview. All figures in the worked example are derived from the synthetic assets/starter-data/ pack and are illustrative placeholders, not Top Spirit (Marussia) data.

0. What this file is

This is the concrete, buildable version of the planning sketch in power-bi-desktop-pilot-pack.md. It carries the report from idea to something the BICC can build, refresh and explain on their own.

Because no .pbix authoring tool is available in this environment, the report is delivered as:

  1. This build pack — the exact data model, DAX measures and page layout to reproduce the .pbix.
  2. A rendered report — outputs/wave1/power-bi-pilot-net-sales-report.md shows what each page displays, with worked figures from the starter pack.
  3. A lineage file — outputs/wave1/power-bi-pilot-lineage.csv traces every figure back to its source export.

Report output ID: OUT-PBI-W1-001 — Top Spirit Wave 1 Sales & Depletions Pilot.

Acceptance-criteria coverage: AC#1 §2–§3 (export-only model), AC#2 §4 (net sales, GP, depletions visuals), AC#3 §7 (BICC refresh & explain runbook).

1. Scope of the pilot report

One report, three questions a steering-committee or commercial reader can answer at a glance:

  • How much did we sell (net)? — net sales by period, channel and category.
  • How profitable was it? — gross profit and GP %, on an explicitly placeholder cost until the cost export lands.
  • How much product moved? — depletions in bottles by channel and category.

Deliberately out of scope for Wave 1: AR/cash position, forecasting, customer-level detail beyond channel/region, and anything needing the Power BI Service.

2. Data sources (export-only — AC#1)

Connect Power BI Desktop with the Folder connector (or SharePoint Folder against the live Export Layer). No database, gateway or live ERP connector is used.

Table Source export ID Starter file Grain Classification
sales_invoices (fact) EXP-ERP-SALES-001 assets/starter-data/sales_invoices.csv invoice line Restricted
customer_master (dim) EXP-CRM-CUST-001 assets/starter-data/customer_master.csv customer Confidential
price_list (dim) EXP-ERP-PRICE-001 assets/starter-data/price_list.csv product code Internal
stock_availability (ref) EXP-WMS-STOCK-001 assets/starter-data/stock_availability.csv product/warehouse Internal
Date (dim) CALC-DATE-001 generated in Power BI date n/a
cost_price (dim) EXP-ERP-COST-001 — NOT YET SUPPLIED placeholder column product code Restricted

Open data dependency: there is no cost export in the starter pack. GP and GP % below run on an illustrative placeholder cost and must not be shown to the client as profit until EXP-ERP-COST-001 is supplied and the Finance SME confirms the cost basis (standard vs landed).

3. Data model (AC#1)

A simple star schema — one fact, three dimensions, one reference table.

            customer_master (EXP-CRM-CUST-001)
                   |  customer_id
                   |
  Date  ----<  sales_invoices (EXP-ERP-SALES-001)  >----  price_list (EXP-ERP-PRICE-001)
 (CALC)   invoice_date          product_code                   product_code
                                                                   |
                                                          stock_availability
                                                          (EXP-WMS-STOCK-001)

Relationships (all single-direction, one-to-many from dimension to fact):

From To Key Cardinality
customer_master sales_invoices customer_id 1 → *
price_list sales_invoices product_code 1 → *
Date sales_invoices invoice_date 1 → *
price_list stock_availability product_code 1 → *

Date is generated with CALENDARAUTO() and marked as the date table. Fiscal-year handling is TBD with Finance (calendar year assumed in the pilot).

4. Core measures (AC#2)

DAX written against the model above. Every measure links to a Business Glossary term and an audit-trail lineage ID.

Measure DAX Glossary / lineage
Net Sales Net Sales = SUM ( sales_invoices[net_amount_eur] ) Net Sales / CALC-NETSALES-001
Depletions (bottles) Depletions = SUM ( sales_invoices[quantity] ) Depletions / CALC-DEPLETIONS-001
Gross Profit (illustrative) Gross Profit = [Net Sales] - [Total Cost] Gross Profit / CALC-GP-001
GP % (illustrative) GP % = DIVIDE ( [Gross Profit], [Net Sales], 0 ) Gross Profit % / CALC-GPPCT-001
Total Cost (placeholder) Total Cost = [Net Sales] * 0.60 CALC-COST-PLACEHOLDER-001

Placeholder, not a client figure. Total Cost uses a flat illustrative 60% cost-to-net ratio purely so the GP and GP % visuals render with a realistic shape. The moment EXP-ERP-COST-001 arrives, replace Total Cost with: Total Cost = SUMX ( sales_invoices, sales_invoices[quantity] * RELATED ( cost_price[cost_price_eur] ) ) and re-run the reconciliation in §7 before anyone treats GP as real.

Definitions still owned by humans, to confirm before adoption:

Measure Open question Owner
Net Sales Are credit notes and rebates already netted in net_amount_eur, or held in a separate export? Finance SME
Gross Profit Standard cost or landed cost? Which cost export is authoritative? Finance SME
Depletions Is "depletions" sell-out (warehouse → trade) or sell-in? Confirm against logistics definition. Logistics / BICC

6. Build steps (reproduce the .pbix)

  1. Open Power BI Desktop (free; covered by E3 — no Pro needed for desktop authoring).
  2. Get Data → Folder → point at assets/starter-data/ (or the live Export Layer SharePoint folder).
  3. Load sales_invoices, customer_master, price_list, stock_availability. Set data types (dates as Date, amounts as Decimal, quantity as Whole Number).
  4. Modeling → New Table → Date = CALENDARAUTO(); mark as date table; build the relationships in §3.
  5. Add the measures in §4 (start with the placeholder Total Cost).
  6. Build pages 1–5 per §5.
  7. Add a visible "Last refreshed" card (= NOW()) and a "Cost = ILLUSTRATIVE PLACEHOLDER" text box on every page showing GP.
  8. File → Save as TopSpirit_Pilot_W1.pbix locally; share via the project SharePoint/Teams. Do not publish to the Power BI Service (needs Pro/PPU — see §8).

7. BICC refresh & handover runbook (AC#3 — refresh and explain independently)

The pilot only counts as delivered when the BICC can refresh it and explain it without us in the room. This runbook is the handover.

7a. How to refresh (BICC, unaided)

  1. Drop the new export files into the agreed Export Layer folder, keeping the exact file names (sales_invoices.csv, etc.).
  2. Open TopSpirit_Pilot_W1.pbix → Home → Refresh.
  3. Confirm the "Last refreshed" card shows today and there are no refresh errors.
  4. Eyeball the Source & lineage page: do the source dates match the exports you just dropped?

7b. Reconcile before trusting (mandatory, human-in-control)

Check Pass condition Reviewer
Net Sales total vs current trusted report Within agreed tolerance (e.g. ±0.5%) of RPT-BICC-SALES-001 Finance SME — DEC-BICC-001
Depletions total vs logistics figure Matches warehouse outbound for the period Logistics / BICC
GP shown? Only after EXP-ERP-COST-001 is loaded and the placeholder is removed Finance SME
Row counts Invoice-line count matches the export row count BICC

Record the outcome in the Source-linked Audit Trail (docs/05-phase-0/source-linked-audit-trail-standard.md) as a DEC- sign-off. No figure is "trusted" until a named human signs the reconciliation.

7c. "Can the BICC explain it?" — handover checklist

The BICC lead should be able to state, unprompted:

  • [ ] Which export feeds each table, and where the folder lives.
  • [ ] The plain-English definition of Net Sales, GP and Depletions, and who owns each.
  • [ ] That GP is on a placeholder cost until the cost export arrives.
  • [ ] How to refresh and how to tell a refresh failed.
  • [ ] Where a wrong figure gets logged (audit trail) and who approves a fix.

7d. Known limitations (carry into Wave 2)

  • Cost/GP is illustrative until EXP-ERP-COST-001 lands.
  • Starter data is a single week and six invoice lines — trend and seasonality are not yet meaningful.
  • No customer-level PII is shown by design; any drill to customer name needs a GDPR review (docs/05-phase-0/privacy-gdpr-operating-model.md).
  • Calendar vs fiscal period is assumed calendar; confirm with Finance.
