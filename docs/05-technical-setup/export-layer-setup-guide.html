M365 E3 Export Layer technical setup guide

How to land, organise, secure and consume file exports inside the existing Microsoft 365 E3 tenant.

Project-team working preview. Aligned with the six non-negotiables: export-layer first, no new licences, human oversight, privacy, knowledge transfer, iterative delivery.

1. What the Export Layer is

The Export Layer is a set of SharePoint/OneDrive folders, Excel/CSV/JSON files, and Power Automate flows that sits between source systems and the intelligence layer.

  • Source systems continue to produce their normal exports.
  • The Export Layer receives, validates, versions and classifies those exports.
  • AI agents, Power BI Desktop reports and the Business Glossary all read from the Export Layer - never directly from source systems unless a business case justifies a connector.

This keeps the engagement ERP-agnostic and lets it survive the pending ERP and BI decisions.

2. Recommended SharePoint site structure

Create one SharePoint site per engagement, then libraries/folders underneath:

Top Spirit  -  Angel Engagement (SharePoint site)
├── Export Layer (document library)
│   ├── raw/                     # untouched source exports
│   ├── validated/               # schema-checked, renamed exports
│   ├── archive/                 # previous versions and monthly snapshots
│   └── schemas/                 # expected headers and validation rules
├── Agents (document library)
│   ├── inputs/                  # files queued for AI processing
│   ├── drafts/                  # AI-generated drafts awaiting approval
│   └── outputs/                 # approved outputs ready for downstream use
├── Audit (document library or list)
│   └── source-linked-audit-trail.csv
├── Glossary (list)
│   └── Business Glossary terms, owners, definitions
└── Opportunity Catalogue (list)
    └── opportunity cards and business cases

OneDrive can be used as a personal drop zone for SMEs who manually export files, with a Power Automate flow moving them to the SharePoint raw/ folder.

3. Naming conventions

Use consistent file names so flows, reports and audit logs can parse them:

<system>_<export-name>_<YYYY-MM-DD>[_<sequence>].<ext>

Examples:

erp_sales-invoices_2026-06-22.csv
crm_customer-master_2026-06-22.xlsx
pricing_active-price-list_2026-06-22.csv
bank_payments-received_2026-06-22.csv

Classification suffix (optional but recommended):

_confidential
_internal
_restricted

4. Power Automate flows

4.1 Ingestion flow

Trigger: file created in OneDrive/SharePoint drop zone.

Steps:

  1. Virus scan / malware check (defender-aware endpoint policy).
  2. Move to Export Layer/raw/<system>/.
  3. Extract metadata: filename, date, size, uploader.
  4. Apply sensitivity label based on file name or source system.
  5. Log receipt in the Source-linked Audit Trail.

4.2 Validation flow

Trigger: scheduled (e.g., daily at 06:00) or manual.

Steps:

  1. List files in raw/<system>/ newer than last run.
  2. Check file extension, encoding and expected headers against schemas/.
  3. Move valid files to validated/; move invalid files to raw/quarantine/ with notification.
  4. Append validation result to audit log.

4.3 Agent processing flow

Trigger: new file in Agents/inputs/.

Steps:

  1. Identify use case from filename or metadata.
  2. Run the relevant AI agent prompt.
  3. Write draft to Agents/drafts/.
  4. Open an Approval Envelope (Power Automate approval or Teams adaptive card).
  5. On approval, copy output to Agents/outputs/; on rejection, route back to draft with feedback.
  6. Record every step in the audit trail.

5. Permission model

Map Microsoft 365 groups to the data-classification tiers:

Group Access Examples
AI Steering Committee Read all + approve Tier 3 Executive sponsor, finance director
BICC / BI lead Read/Write Export Layer + Audit + Glossary Technical counterpart
Process owners Read/Write relevant validated/ folders + their Approval Envelopes CS team lead, finance controller
SMEs / export owners Write to drop zone + read their own exports Warehouse, pricing assistant
Agents (service account) Read validated inputs + Write drafts + Write audit log Power Automate service account

Principles:

  • Least privilege: users see only the exports their role needs.
  • No shared personal logins for agents; use a dedicated service account or managed identity.
  • Quarterly access review.

6. Power BI Desktop connection

Power BI Desktop reads CSV/Excel files from SharePoint via the SharePoint Folder connector.

Important: under M365 E3, Power BI Desktop is free to use. Publishing to Power BI Service requires Pro/PPU licences, so the pilot stays in Desktop unless the client already has Pro.

Recommended pattern:

  1. Connect to Export Layer/validated/ as a folder.
  2. Filter by filename pattern (e.g., erp_sales-invoices_*.csv).
  3. Combine files and add a loaded_date column from filename or file metadata.
  4. Build measures that reference the Business Glossary.
  5. Save the .pbix locally and share via the project Teams/SharePoint.

7. Retention and versioning

  • Enable SharePoint versioning on Export Layer/ and Agents/ (major + minor).
  • Retain raw exports for 13 months unless a longer period is required by finance/tax.
  • Archive monthly snapshots to archive/YYYY-MM/.
  • Audit log: append-only CSV or SharePoint list; retain for the engagement lifetime + 2 years.

8. Security checklist

  • [ ] Sensitivity labels applied to Confidential and Restricted folders.
  • [ ] DLP policy prevents downloading Restricted exports to unmanaged devices.
  • [ ] MFA required for all engagement site access.
  • [ ] Agent service account has least privilege and is excluded from interactive logon.
  • [ ] No premium connectors used unless business case approved.
  • [ ] Guest access disabled unless explicitly required.
